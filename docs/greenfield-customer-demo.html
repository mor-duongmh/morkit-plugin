Step 6 · Bridge — gom thành dữ liệu

Biến cuộc trò chuyện thành dữ liệu có cấu trúc

Năm bước đầu là hỏi-đáp và quyết định. Bước 6 gom tất cả vào một bảng dữ liệu gốc — file mà máy đọc để tự sinh ra SRS và tài liệu thiết kế. Đây là cầu nối giữa "nói chuyện với người" và "tài liệu hoàn chỉnh".

Một chức năng trông như thế này trong file:

"id": "FR-001", "name": "Hỏi-đáp tri thức (chat)", "priority": "High", "related_uc": ["UC-001"], "source": "brainstorm-report.md"

Đọc từng dòng:

  • id · name — mã và tên chức năng
  • priority — độ ưu tiên (High / Mid / Low)
  • related_uc — gắn với tình huống dùng (use-case) nào
  • source — lấy từ đâu: chức năng này đến từ brainstorm-report.md, lần ngược về đúng câu bạn đã chốt ở Bước 2

Vì sao quan trọng: trường source bắt mỗi mục phải có gốc. Thiếu nguồn thì đánh dấu <TBD> — máy không tự bịa. File được kiểm hợp lệ tự động (13 chức năng, 2 use-case) trước khi sinh tài liệu.

Step 6 · Bridge — gather into data

Turn the conversation into structured data

The first five steps are Q&A and decisions. Step 6 gathers everything into one source data table — the file the machine reads to auto-generate the SRS and design docs. It's the bridge between "talking to a person" and "a finished document".

One function looks like this in the file:

"id": "FR-001", "name": "Knowledge Q&A (chat)", "priority": "High", "related_uc": ["UC-001"], "source": "brainstorm-report.md"

Reading line by line:

  • id · name — function code and name
  • priority — priority (High / Mid / Low)
  • related_uc — which use-case it ties to
  • source — where it came from: this function comes from brainstorm-report.md, traceable back to exactly what you locked at Step 2

Why it matters: the source field forces every item to have an origin. No source → mark it <TBD> — the machine won't make things up. The file is auto-validated (13 functions, 2 use-cases) before any doc is generated.
